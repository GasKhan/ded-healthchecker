# ded-healthbot

Telegram-бот мониторинга health-эндпоинтов сервисов. Опрашивает все
сервисы из `services.yml` с заданным интервалом, шлёт алерты в TG только
при смене статуса (UP↔DOWN). Список получателей управляется командами
владельца — без правки `.env` и рестарта.

## Возможности

- Опрос произвольного списка сервисов из `services.yml`.
- Per-service настройки: `interval`, `timeout`, `retries`, `retry_delay`.
- Защита от ложных DOWN: 429 трактуется как «жив, нас тротлят», а не как
  падение.
- Алерт **только при смене статуса** — никакого спама.
- Динамический список подписчиков (`/subscribe`, `/unsubscribe`).
- Команда `/status` — снимок всех сервисов.
- Авто-удаление чата из подписчиков при `TelegramForbiddenError` (бот
  кикнут или забанен).

## Стек

- Python 3.13 (через pyenv)
- poetry
- aiogram 3.x
- aiohttp
- pydantic + pydantic-settings
- PyYAML, aiofiles

## Установка

```bash
git clone <repo> ded-healthbot
cd ded-healthbot
pyenv install 3.13     # если ещё нет нужной версии
pyenv local 3.13
poetry install
cp .env.example .env   # заполнить BOT_TOKEN и ADMIN_CHAT_ID
```

## Конфигурация

### `.env`

| Переменная       | Назначение                                                                |
|------------------|---------------------------------------------------------------------------|
| `BOT_TOKEN`      | Токен бота от @BotFather                                                  |
| `ADMIN_CHAT_ID`  | chat_id чата-владельца (личка или группа админов). Команды управления принимаются только оттуда. |

### `services.yml`

```yaml
defaults:
  interval: 60
  timeout: 10
  retries: 3
  retry_delay: 5

services:
  - name: dedai-prod
    url: https://api.dedai.ru/health
```

`name` и `url` обязательны. Любой параметр из `defaults` можно
переопределить на уровне сервиса.

## Запуск

```bash
poetry run python -m healthbot
```

## Команды бота

| Команда                       | Кто может       | Что делает                                        |
|-------------------------------|------------------|---------------------------------------------------|
| `/status`                     | подписчики       | Снимок состояния всех сервисов                    |
| `/list`                       | подписчики       | Список настроенных сервисов из `services.yml`     |
| `/subscribe`                  | ADMIN_CHAT_ID    | Добавить **текущий** чат в подписчики             |
| `/subscribe <chat_id>`        | ADMIN_CHAT_ID    | Добавить произвольный chat_id                     |
| `/unsubscribe`                | ADMIN_CHAT_ID    | Удалить текущий чат                               |
| `/unsubscribe <chat_id>`      | ADMIN_CHAT_ID    | Удалить произвольный chat_id                      |
| `/recipients`                 | ADMIN_CHAT_ID    | Показать список получателей                       |

## Логика статусов

- При старте все сервисы — `UNKNOWN`.
- Первая успешная проверка → `UP` (без алерта, это норма).
- Первая неудачная (после исчерпания retries) → `DOWN` (с алертом).
- Смена `UP→DOWN` → алерт ❌ с описанием ошибки.
- Смена `DOWN→UP` → алерт ✅ с latency и downtime.
- 429 от опрашиваемого сервиса → не считаем DOWN, пропускаем тик.

Состояние `last_status` хранится **в памяти** и сбрасывается при
рестарте бота. Это сделано намеренно — после рестарта возможен один
лишний `✅` если сервис лежал, что приемлемо.
